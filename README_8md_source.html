4 ## Setup
5 For this course, you need a modern C++ compiler that supports C++20. I recommend using `g++` or `clang`. `MSVC` should
6 also work, but I haven't tested it yet. Furthermore, you need `cmake` (at least version 3.14). And finally, I recommend
7 that you install one of the following libraries on your system: `libdw` (`libdw-dev`), `libdwarf` (`libdwarf-dev`).
8 These are not necessary but help producing meaningful stacktraces which makes debugging easier. I have no idea whether
9 these libraries are available for Windows...
10 
11 All the remaining dependencies are handled by cmake.

102 ## Project Structure
103 The goal of this course is to develop a simple SAT-solver in C++. Don't worry if you're not very familiar with this
104 language. This template should help you focus on the logic of the solver instead of worrying about implementation
105 details. On the other hand, if you're a C++ pro then feel free to deviate from the template as you see fit.

110 ### Implementation of the Solver
111 The solver will be developed as a C++-class in `Solver/Solver.hpp(cpp)`. As a reminder, in C and C++ we usually *declare*
112 functions and classes in the header file. A declaration involves only the function signatures and the class members.
113 The actual *implementation* will be done in the `.cpp` file. Exceptions are template classes and functions, but normally
114 you don't need to worry about that too much. To facilitate the implementation of the solver, we'll make use of the
115 following constructs:
116 * Variable and Literal (in `Solver/basic_structures.hpp(cpp)`)
117 * Clause (in `Solver/Clause.hpp(cpp)`)
118 
119 Additionally, I already implemented some helping constructs in `Solver/util` as well as functions for reading and
120 writing SAT-formulas in `Solver/inout.hpp(cpp)`.

139 ### 1. Basic Structures
140 First we will focus on the implementation of the structures that we use for representing the problem: Variable and
141 Literal. A Variable is more or less just an ID identifying a variable in the SAT-formula. A Literal represents one of
142 two values that a variable can take. Since the variables are boolean, the only possible values for let's say a variable
143 `x` are `x` and `¬x`.
144 
145 Follow the documentation in `Solver/basic_structures.hpp` to implement all the necessary functions. Remember that the
146 implementation goes in the file `Solver/basic_structures.cpp` (simply replace all the method bodies).
147 
148 Test your implementation by building and running the test target `test_basic_structures`.
149 
150 ### 2. Clause
151 Next we need a structure to represent clauses. This can be a simple list that holds a variable number of literals.
152 I recommend using [`std::vector`](https://en.cppreference.com/w/cpp/container/vector) from the header `<vector>`.
153 This is a variable sized array that holds elements in a contiguous block of memory which makes it very efficient in
154 loops and also provides efficient random access. Now you have two choices:
155 1. If you want to implement the unit propagation algorithm using counters, then you don't need to implement anything.
156  Simply use `std::vector<Literal>` as your clause. I recommend that you create a type alias as follows
157  ```c++
158  using Clause = std::vector<Literal>;
159  ```
160  This alias is actually necessary because the tests depend on it. On top of that, it makes your code mor readable.
161  If you choose this option, you can simply delete the class `Clause` in `Solver/Clause.hpp` and remove all the
162  implementations from `Solver/Clause.cpp` (if you delete the `.cpp` file, you need to reload CMake). In this case we
163  don't care that the unit tests for the clause class don't work. You can simply delete `Tests/test_clause.cpp` and
164  reload CMake.
165 2. If you want to implement the more efficient unit propagation using watch literals, I recommend that you use the
166  `Clause`-class already declared in `Solver/Clause.hpp`. Again, you will need to implement all functions in the
167  corresponding `.cpp` file. The class is designed to be immutable. You are not supposed to change a SAT-`Clause` once
168  it has been declared (you cannot add or remove literals). Follow the documentation already provided and test your
169  code using the test target `test_clause`.
170 
171 ### Solver
172 No we get to the actual solver. Depending on the approach that you choose for the unit propagation you need to organize
173 your Clauses and Literals. In any case, you additionally need to keep track of the current variable assignment, i.e.
174 the *model*. This means that you need to save the truth value of every variable. A variable can either be `true`,
175 `false` or `unassigned`. Independently of your unit propagation algorithm you need to be able to efficiently look up
176 clauses by literals. Since you don't want to have multiple copies of the same clauses in your solver, you need to
177 organize them using pointers. I recommend that you use [`std::shared_ptr`](https://en.cppreference.com/w/cpp/memory/shared_ptr)
178 for that purpose. I already made the type aliases `ClausePointer` and `ConstClausePointer` for you in
179 `Solver/Solver.hpp`. The advantage with [`std::shared_ptr`](https://en.cppreference.com/w/cpp/memory/shared_ptr) is that
180 you don't need to worry about memory management.
181 
182 1. Start by declaring all your data structures within the solver class (organize your clauses, unit literals and model
183  etc...). The start with the constructor and methods like `val`, `satisfied`, `falsified`, ...
184  You can run the test target `test_solver` even if you haven't yet finished implementing all methods.
185 2. Implement the **unit propagation** using the algorithm of you choice. This function is crucial for the SAT-solver,
186  make sure it works as intended! Verify your implementation using the simple unit propagation test cases in
187  `test_solver` and using the extra tests in `test_unit_propagation`.
188 3. Once your unit propagation works, you can finally implement the actual solve loop. Start with the DPLL algorithm.
189  You will see that at some point you will choose the next variable to decide. You can use the `FirstVariable`
190  heuristic from `Solver/heuristics.hpp`. This is probably the simplest heuristic there is (and it is not very good).
191  You can also write your own heuristic (now or later). You don't need to use this class and instead implement your
192  heuristic however you like (for example directly inside the solver). The advantage of an external heuristic class
193  with a well-defined interface is that you can easily replace it later with something more powerful.
194 4. To test your solver, write a simple executable `solve.cpp` that you place in the main directory. You first need
195  to load a problem using the functions from `Solver/input.hpp`. Then you configure your solver, run the search and
196  finally print the result. When printing your solver output, please respect the following rules:
197  * If the instance is UNSAT, print the word `UNSAT`.
198  * If the instance is SAT, print the solution (all unit literals) in DIMACS format. Use the function
199  `sat::inout::to_dimacs` from ´Solver/inout.hpp´ and [`std::cout`](https://en.cppreference.com/w/cpp/io/cout).
200  * Make sure to respect these rules. You can print other stuff as you like as long as you prepend it with the
201  character `c` to mark the line as comment.
202 5. Once your DPLL search works you can:
203  * Implement a better value selection heuristic. If you used the `FirstVariable` heuristic in step 4 I recommend that
204  you use this interface to create your own heuristic. Simply make a copy of `FirstVariable` in
205  `Solver/heuristics.hpp` and don't forget the implementation in the `.cpp` file. You could for example try
206  implementing a heuristic that randomly selects an unassigned variable (there is a random number generator in
207  `Solver/util/random.hpp`).
208  * Implement restarting.
209  * Implement clause learning and ultimately CDCL search. This algorithm is much more powerful than DPLL and should be
210  able to resolve even big problem instances.
211 
212 Note: You will get a good grade if you manage to correctly implement DPLL, no need to do all the tasks in 5. But you
213 will get an even better grade if you put in some extra work :).

215 ## Grading
216 You should hand in a working solver leveraging at least the DPLL algorithm. Create a simple executable as described in
217 step 4 above. Your executable needs to accept at least one command line argument (the path to the problem instance).
218 If the solver accepts more arguments, please document them and give them meaningful default values. The solver needs to
219 be callable with just the problem instance. When printing your solution, respect the rules also detailed in step 4.
220 If you don't, you will lose points!
221 
222 Your solver will be run on a set of different problem instances of different difficulties (the ones under `eval`).
223 The more instance you can solve, the better. Your solver will be given a timeout of 2 minutes for each instance. When
224 running your solver, your code will be built in release mode (with all optimizations) to make it faster. If you want
225 to see how fast your solver can go, you too should build in release mode.
226 
227 Your solver should at least be able to solve the medium instances (both SAT and UNSAT). If it can solve the harder once,
228 that's great. But above all, your solver should not crash. Make sure to write proper and safe code!
